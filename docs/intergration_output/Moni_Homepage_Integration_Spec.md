## 2. 首页总页面边界

### 2.1 首页的产品职责

首页不是传统报表页，而是 Moni 的主舞台。首页承担四类核心职责：

1. 让用户第一眼看懂当前消费状态
2. 让用户以最小摩擦浏览流水并修正分类
3. 让用户感知 AI 是否在工作、学到了什么、是否可控
4. 作为设置页与记账页之外的主入口，不承担复杂配置编辑

### 2.2 首页不承担的职责

以下内容不得塞回首页主界面：

1. 账本管理
2. AI 配置细项
3. 标签体系管理
4. AI 记忆编辑
5. 全量重分类发起配置
6. 预算规则编辑

上述能力归属设置页，其中：

- 全局区：AI 配置 / 自述 / 账本管理 / 关于
- 账本区：标签管理 / AI 记忆 / 预算设置 / 重新分类 / 学习设置

### 2.3 本轮集成不展开的内容

以下内容可能与首页相关，但不属于本轮首页集成实施范围：

1. AI 记忆查看 / 编辑页的新表现层原型
2. 设置页整体重做
3. 手动记录条目的持久化层与学习链路落地
4. 情景提示引擎本体的完整定义
5. AI 引擎配置页逻辑重做

### 2.4 首页组件树

当前原型代码中，首页主要由以下组件组成：

1. 页面壳层：`MoniHomePrototype.jsx`
2. 背景装饰：`Decor`
3. Header：`Logo` + 首页账本选择器
4. 顶部看板：`DisplayBoard`
5. 情景提示：`HintCard`
6. 统计摘要：`StatsBar`
7. 分类概览：`OverviewCard`
8. 时间范围选择：`DateRangeDialog`
9. 分类标签轨道：`TagRail`
10. 按天流水卡：`DayCard`
11. 长按拖拽蒙版：`DragOverlay`
12. 分类后理由输入：`ReasonDialog`
13. 底部导航与 AI 控制：`BottomNav`

## 3. 首页全局状态模型

### 3.1 前端需要持有的核心状态

首页至少需要消费并维护以下状态：

1. `currentLedger`
2. `homeDateRange`
3. `selectedCategoryFilter`
4. `displayBoardMode`
5. `hintCards`
6. `dailyTransactionGroups`
7. `unclassifiedCount`
8. `aiEngineUiState`
9. `dragSessionState`
10. `dateRangeDialogState`

### 3.2 推荐的 AI 引擎 UI 状态机

首页 UI 层不应只有简单的 `aiOn: boolean`。联调口径建议拆为以下有限状态：

- `idle`：空闲
- `running`：正在消费当前账本的队列任务
- `draining`：收到关闭指令后，正在完成当前批次
- `paused`：引擎已关闭，不继续消费
- `error`：引擎控制或消费出现异常

补充字段建议：

- `activeLedger`
- `activeDate`
- `hasPendingInRange`
- `hasPendingOutOfRange`
- `pendingCount`
- `lastLearnedAt`
- `lastLearningNotice`

### 3.3 表现层 / 逻辑层事件口径

首页联调建议统一使用“读模型 + 动作”的模式：

1. 表现层读取聚合后的首页读模型
2. 用户操作通过显式逻辑层动作发起
3. AI 工作态、学习提示、范围外 backlog 提示通过状态订阅、事件总线或轮询刷新

## 4.3 Header：Logo + 账本选择器

### 已定稿规格

1. 左侧固定为完整 `Moni` 字标
2. 右侧为账本选择器
3. 默认账本为“日常开销”
4. 首页账本选择器只负责切换，不负责复杂管理
5. 账本管理入口归属设置页全局区
6. Header 为粘性顶部元素

### 业务逻辑规格

1. Header 进入首页即显示当前账本名称
2. 点击账本选择器，只打开“账本切换表单 / 下拉 / 底部面板”
3. 选择新账本后，首页整体切换到目标账本上下文
4. 若切换离开当前账本时该账本处于“待学习”，允许逻辑层按 v6 规格静默触发学习会话
5. 首页不提供新增、重命名、删除账本操作

### 推荐读模型 / 动作口径

#### 表现层读取

首页上下文读模型建议至少包含：

```json
{
  "currentLedger": {
    "id": "ledger_daily",
    "name": "日常开销"
  },
  "availableLedgers": [
    { "id": "ledger_daily", "name": "日常开销" },
    { "id": "ledger_trip", "name": "旅行账本" }
  ]
}
```

#### 逻辑层动作

`switchLedger(ledgerId)`

### 联调规则

1. 切账本后应刷新首页聚合数据、AI 状态、时间范围内流水
2. 切账本期间需要短暂的 loading / skeleton，防止旧账本数据闪烁
3. 如果目标账本没有数据，首页应进入空账本态，而不是保留旧数据

### 明确冻结的业务边界

- 首页账本选择器只能是“选择表单”
- 账本管理必须放在设置页，不得在首页下拉中继续膨胀

### 当前现状

- UI 已展示账本名与下拉箭头
- 仅静态显示“日常开销”
- 没有真实切换动作，也没有设置页跳转入口

### 待沉淀 / 待补进 DESIGN 的点

- 首页账本切换面板的具体形态可继续后续定，但其业务边界已明确：只切换，不管理

## 4.4 顶部看板 DisplayBoard

### 已定稿规格

1. 看板是显示卡容器，不是复杂分析区
2. 有预算时显示预算卡 + 近 7 天折线图卡
3. 无预算时只显示折线图卡
4. 切换方式为上下切换
5. 自动轮播周期 30 秒
6. 手动切换后暂停自动轮播 5 分钟
7. 折线图卡内部左右切换时间窗口

### 业务逻辑规格

#### 预算卡

展示字段：

- `budgetPeriodLabel`
- `budgetAmount`
- `spentAmount`
- `remainingAmount`
- `remainingDays`
- `dailyAvailableAmount`
- `budgetStatus`

#### 折线图卡

展示字段：

- `trendWindowLabel`
- `points`
- `windowStart`
- `windowEnd`
- `hasMoreHistoryBefore`
- `hasMoreHistoryAfter`

### 推荐读模型口径

首页聚合读模型中直接返回：

```json
{
  "displayBoard": {
    "hasBudget": true,
    "budgetCard": {
      "periodLabel": "4月预算",
      "budgetAmount": 3128,
      "spentAmount": 1256,
      "remainingAmount": 1872,
      "remainingDays": 24,
      "dailyAvailableAmount": 78,
      "status": "healthy"
    },
    "trendCard": {
      "windowSize": 7,
      "points": [
        { "date": "2026-04-01", "amount": 120 },
        { "date": "2026-04-02", "amount": 95 }
      ],
      "hasEarlierWindow": true,
      "hasLaterWindow": false
    }
  }
}
```

### 联调规则

1. 看板展示使用首页当前 `homeDateRange`，但折线图默认展示近 7 天窗口
2. 若预算未配置，预算卡彻底消失，圆点同步消失
3. 折线图切窗属于表现层交互，也可以由逻辑层直接按窗口返回切片数据
4. 首页不承担预算编辑，预算配置入口仍归设置页账本区

### 当前现状

- 上下轮播、自动轮播、折线图横滑已实现
- 数据全部来自 mock
- 预算状态颜色逻辑已实现简版
- 未接真实预算、趋势、历史窗口数据

### 待联调点

- 逻辑层需明确预算缺省态
- 趋势图是否一次返回全量点集，还是按窗口分页返回，需要统一

## 4.5 情景提示卡 HintCard

### 已定稿规格

1. 位于看板与统计摘要之间
2. 有内容时显示，无内容时整块消失
3. 不是弹窗，不打断阅读

### 已明确的内容类型

1. 距上次导入已 N 天
2. 本周消费偏高
3. AI 学到新的偏好
4. 首次导入后对 AI 控制入口的提示
5. 当前范围外仍有待处理任务

其中第 5 项来自 v6 队列调度规格，虽未写进 `DESIGN.md` 首页章节，但应作为首页提示卡候选能力。

### 业务逻辑规格

提示卡应由逻辑层或状态聚合层返回结构化提示列表，而不是表现层硬编码文案。

推荐结构：

```json
{
  "hintCards": [
    {
      "id": "import_gap_7d",
      "type": "import_reminder",
      "priority": "medium",
      "title": "距上次导入已 7 天",
      "description": "导入新账单看看最近花了多少？",
      "primaryAction": {
        "type": "navigate",
        "target": "bookkeeping_import"
      },
      "dismissible": true
    }
  ]
}
```

### 联调规则

1. 首页同一时刻建议只展示一张主提示卡，避免视觉噪音
2. 提示卡可关闭，但关闭策略要区分“仅本次隐藏”还是“持久忽略”
3. 与 AI 学习成功相关的轻量通知，可采用顶部 toast；若需要承接更多操作，可沉淀为提示卡

### 当前现状

- 只实现了单一 mock 提示卡
- 关闭后仅本地隐藏
- 导入按钮没有真实动作

### 待沉淀点

- “AI 学到了新偏好”在聊天和 v6 中已明确，但首页用 toast 还是提示卡，仍需统一到 `DESIGN.md`
- “范围外 backlog 提示”已在 v6 冻结存在性，但首页视觉形态尚未在 `DESIGN.md` 定稿

## 4.6 统计摘要栏 StatsBar

### 已定稿规格

1. 固定为三张小卡：支出 / 收入 / 笔数
2. 标题必须与 Data Range 联动
3. 只承担快速理解，不做深度分析

### 业务逻辑规格

输入字段：

- `rangeLabel`
- `expenseTotal`
- `incomeTotal`
- `transactionCount`

命名规则：

- 快捷范围：`本月支出`、`本周收入`、`今日支出`
- 自定义范围：`区间支出`、`区间收入`、`区间笔数`

### 推荐读模型口径

```json
{
  "statsBar": {
    "rangeLabel": "本月",
    "expenseTotal": 847.5,
    "incomeTotal": 5800,
    "transactionCount": 46
  }
}
```

### 当前现状

- 已实现三卡展示与自定义范围命名差异
- 数据来自 mock 计算

### 待联调点

- 需要统一支出 / 收入统计是否都受当前账本和当前范围约束
- 需要确认是否包含内部转账、退款、失败交易等过滤逻辑

## 4.7 分类概览卡 OverviewCard

### 已定稿规格

1. 使用横向堆叠条，不使用饼图
2. 图例与横条一一对应
3. 保留所有已出现分类
4. 未分类段必须使用半透明或斜杠阴影表达
5. 右上角是 Data Range Picker 唯一入口

### 业务逻辑规格

输入字段建议：

- `categoryBreakdown[]`
  - `category`
  - `amount`
  - `percent`
  - `count`
  - `isUncategorized`

### 推荐读模型口径

```json
{
  "overviewCard": {
    "rangeLabel": "本月",
    "categoryBreakdown": [
      { "category": "正餐", "amount": 320, "percent": 45, "count": 8 },
      { "category": "交通", "amount": 140, "percent": 20, "count": 5 },
      { "category": "未分类", "amount": 60, "percent": 8, "count": 2 }
    ]
  }

```

### 联调规则

1. 分类概览只展示当前范围内已出现的分类
2. 未分类必须单独统计
3. 分类名称直接使用中文键，不单独维护显示映射
4. 点击右上角只负责打开 Data Range Picker，不承担更多操作

### 当前现状

- 视觉规则已基本对齐
- 数据由 mock 计算

## 4.8 时间范围面板 DateRangeDialog

### 已定稿规格

1. 入口只有分类概览右上角
2. 快捷项固定为：今天 / 本周 / 本月 / 近三月
3. 支持自定义范围
4. 自定义范围保留双档位滑条 + 起止日期输入
5. 拖动时日期实时更新
6. 同时支持鼠标与触控

### 业务逻辑规格

#### 前端职责

1. 管理面板开关
2. 显示当前范围
3. 选择快捷项或自定义范围后触发首页重取

#### 逻辑层职责

1. 返回在该范围下的统计、概览、流水分组
2. AI 引擎消费许可窗口以当前 `homeDateRange` 为准
3. 若当前范围外仍有待处理任务，需要返回对应信号供首页提示

### 关键联调规则

根据 v6：

1. `data range` 只约束 AI Engine 消费，不约束任务生产
2. 首页切换范围后，如果队列中存在范围外任务，不得视为完成
3. UI 应可感知 `hasPendingOutOfRange`

### 推荐动作口径

表现层可仅本地更新范围状态，再触发首页聚合重算或读模型刷新：

- `setHomeDateRange({ type: "month" })`
- `setHomeDateRange({ type: "custom", start: "2026-03-01", end: "2026-04-07" })`

### 当前现状

- 快捷项、自定义范围、双滑块、输入框、遮罩关闭已实现
- 联动到了统计、概览、流水过滤
- 还未接 AI 队列消费许可窗口与 backlog 状态

### 待补进 DESIGN 的点

- `data range` 与 AI 消费调度的关系目前主要在 v6 中冻结，首页层面建议后续补一条引用说明到 `DESIGN.md`

## 4.9 分类标签轨道 TagRail

### 已定稿规格

1. 是横向标签轨道，不是静态分页
2. 原型阶段允许先做有限轨道
3. 激活态深色填充，未激活态白底描边
4. 未分类为警示态
5. 吸附后固定在 Header 下方
6. 分类键当前直接使用中文

### 业务逻辑规格

1. 标签轨道是当前首页流水过滤器
2. 支持：
   - 全部
   - 各分类
   - 未分类
3. 切换后只改变当前首页展示，不直接改数据
4. 标签本身不承担标签管理能力

### 推荐前端输入

```json
{
  "tagRail": {
    "filters": [
      { "key": "全部", "label": "全部", "count": 46 },
      { "key": "正餐", "label": "正餐", "count": 8 },
      { "key": "未分类", "label": "未分类", "count": 2, "warning": true }
    ],
    "selectedKey": "全部"
  }
}
```

### 当前现状

- 已实现横向滚动、激活态、未分类计数、sticky 吸附
- 分类源自本地常量

## 4.16 AI 引擎控制条

### 已定稿规格

1. 长按中央按钮后向上弹出纵向开关条
2. 手指保持按住并滑动完成选择
3. 不是两个独立点击按钮
4. 关闭采用软停止，不是硬中断

### 业务逻辑规格

#### UI 层动作

中央按钮只应发出两个明确意图：

1. `request_start`
2. `request_soft_stop`

不应由 UI 直接决定：

1. 立即中断当前 LLM 请求
2. 清空队列
3. 回滚已完成写入

#### 推荐逻辑层动作

- `requestAiEngineStart({ ledgerId })`
- `requestAiEngineSoftStop({ ledgerId })`

#### 推荐动作结果

```json
{
  "accepted": true,
  "engineState": "running",
  "effectiveMode": "running"
}
```

或

```json
{
  "accepted": true,
  "engineState": "draining",
  "effectiveMode": "soft_stopping",
  "activeDate": "2026-04-04"
}
```

### 状态回写规则

首页 UI 应以逻辑层真实状态为准：

1. 用户请求开启成功后，状态进入 `running`
2. 用户请求关闭后，如果当前正在处理，应进入 `draining`
3. 只有当前批次安全完成后，才从 `draining` 回到 `paused` 或 `idle`

### 与 v6 的对应关系

1. AI Engine 消费的是当前账本队列
2. `data range` 只约束消费许可窗口
3. 当前范围外任务仍要保留
4. 首页应能感知“有范围外待处理任务”

### 当前现状

- 长按弹条、滑动命中、软停止动画已做成原型态
- 实际只是本地状态切换与 `setTimeout`
- 没有真实控制 API

### 明确冻结的动作边界

- UI 只表达“请求开启 / 请求软停止”
- 是否真正启动消费、消费哪个日期、何时完成停止，全部由逻辑层 AI Engine 决定并回写状态

## 8. 推荐联调顺序

1. 先定义首页聚合读模型
2. 再接账本切换与时间范围参数
3. 再接 AI 引擎状态查询 / 订阅
4. 再接拖拽分类动作与可选理由补充
5. 最后接提示卡体系与 backlog 感知

推荐原因：

1. 首页大部分组件依赖同一批聚合数据
2. AI 工作态与队列状态必须先有真实读模型，视觉联动才不会失真
3. 分类修正动作接通后，首页才具备真正的“教 AI”闭环

## 9. 最终结论

当前首页的视觉与交互骨架已经基本定稿，真正缺少的不是更多组件，而是把以下三条业务边界彻底冻结并联通：

1. 首页只做账本切换，不做账本管理
2. AI 控制条只表达 UI 意图，由逻辑层决定真实运行 / 软停止状态
3. 首页所有展示组件都应建立在统一的首页聚合读模型之上，而不是继续各自 mock

当这三条打通后，首页就可以从“高保真原型”平滑过渡到“可联调的正式前端壳层”。
