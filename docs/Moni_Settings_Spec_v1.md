## 0. 文档定位

本文件用于在“设置页可交互原型设计”开始前，把 Moni 当前分散在多份文档、主仓库实现、参考子仓库中的设置相关信息统一收口。

自 **2026-04-09** 起，设置页相关文档若存在冲突，必须以**用户当轮确认口径**与**本文件**为准；其他设置页文档只能做展开说明，不得反向覆盖本文件。

本文件回答三个问题：

1. Moni 现在到底有哪些“设置”
2. 这些设置属于全局还是账本级
3. 哪些已经在主仓库实现，哪些只有文档口径，哪些 UI 仍缺位

本文件是设置页原型设计的参考规格，不替代以下专项文档：

- 预算细节仍以 `docs/Moni_Budget_System_Spec_v2.md` 为准

- 总体产品边界仍以 `docs/Moni_Requirements_v3.md` 为准

## 2. 设置系统总览

Moni 的设置系统不是单一表单，而是一个分层系统。

当前冻结口径：

- 设置页分为 **全局区** 和 **账本区**
- 全局区承接“跨账本共享”的配置
- 账本区承接“当前账本专属”的配置
- 首页只负责消费设置结果，不负责复杂编辑
- 首页顶部账本选择器只负责切换账本，不承担账本管理

### 2.1 全局区

全局区应包含：

- AI 引擎配置 / 模型配置 / 提供方配置
- 自述
- 账本管理
- 数据导出
- 关于

扩展说明：

- 数据导出已在需求文档中列为 P2，更适合放在全局区，但当前未冻结进首轮原型必做范围
- 主题切换在主仓库旧设置页中已有实现，但未进入当前产品规格文档，应视为“现存实现残留能力”，不是当前设置 IA 的核心项

### 2.2 账本区

账本区应包含：

- 标签管理
- AI 记忆
- 预算设置
- 学习设置
- 全量重新分类入口

补充解释：

- “AI 记忆”偏内容编辑与历史版本管理
- “学习设置”偏行为参数，例如学习阈值、自动学习、收编阈值、收编压缩比例
- “重新分类”是动作型设置，不是静态表单项

## 4. 各设置项的正式定义

本章开始，统一使用以下状态口径，避免“已实现”产生歧义：

- **主仓库逻辑已实现**：主仓库已有数据结构、服务接口、持久化或业务链路
- **主仓库新设置页表现层未实现**：面向 Moni 当前阶段目标的设置页表现层尚未正式完成，不能视为已落地
- **旧参考设置页已有实现**：仅指 `src/ui/components/mobile/SettingsPage.tsx` 中存在旧参考型面板，不等于当前正式设置页 IA 已落地

因此，本章凡提到“当前实现结论”，默认优先描述逻辑层状态；若涉及表现层，会明确写“旧参考设置页”或“新设置页表现层”。

## 4.1 全局区

### 4.1.1 AI 配置

AI 配置由以下层级组成：

1. 提供方
2. API Key
3. Base URL
4. 候选模型
5. 全局推理参数

当前主仓库逻辑状态：

- `ConfigManager` 已支持 provider / apiKey / baseUrl / candidateModels / globalParams 的读写
- AI 引擎启动前已会检查 API Key 是否存在

旧参考设置页已有字段：

- provider
- apiKey
- baseUrl
- model

当前已存在但未暴露到正式设置原型的底层参数：

- `globalParams.maxTokens`
- `globalParams.temperature`
- `globalParams.enableThinking`
- `ui.language`

原型设计要求：

- AI 配置需要明确“当前生效模型”
- 自定义 provider 与预设 provider 应分开表现
- 连接测试是高价值动作，应保留
- 不要把“AI 配置”和“AI 记忆 / 学习设置”混在同一卡片

### 4.1.2 自述

自述是全局共享的人类偏好描述，不按账本隔离。

用途：

- 为 AI 补充长期个人偏好
- 作为分类、学习、收编 prompt 的全局上下文

当前实现结论：

- 主仓库逻辑已实现独立读写能力
- 数据已从旧 `secure_config.bin` 内的 `userContext` 迁移到独立文件
- 旧参考设置页已有文本编辑面板
- 主仓库新的正式设置页表现层仍未实现

本轮新增冻结口径：

- 自述页默认文案必须显式呈现 demo 语气，避免让原型看起来像真实用户长期资料
- 默认文案应强调“这里是给 AI 的示例偏好描述，可按自己的真实习惯改写”
- 自述页保存仍为显式提交，不做输入中自动保存

### 4.1.3 账本管理

账本管理属于全局区，但操作对象仍是账本。

应包含的动作：

- 切换账本
- 新建账本
- 重命名账本
- 删除账本

当前实现结论：

- 主仓库逻辑已支持 `create / rename / delete / switch`
- 默认账本不可删除
- 账本重命名和删除会级联迁移或清理 AI 相关文件、预算配置、账本行为配置
- 旧参考设置页只有“当前账本切换”入口，缺少完整 CRUD 面板
- 主仓库新的正式设置页表现层仍未实现

因此，设置页原型设计时，账本管理不能再只保留一个“当前账本”行项，必须设计成完整管理区块。

### 4.1.4 关于

关于页面在需求中存在，但当前没有正式内容结构。

建议原型阶段至少预留：

- 应用名与版本
- 产品定位短文案
- 构建时间或版本号
- 关键文档 / 反馈 / 作者信息入口

### 4.1.5 数据导出

数据导出当前仍是 P2。

原型建议：

- 先保留入口，不展开复杂交互
- 只在规格里注明为后续扩展项

## 4.2 账本区

### 4.2.1 标签管理

标签管理承接账本分类体系主数据。

当前冻结口径：

- 标签键直接使用中文存储键
- 新增标签应附描述
- 标签描述是冷启动锚点、学习锚点和新增门槛
- `defined_categories` 是标签主数据单一信源

当前实现结论：

- 主仓库逻辑已支持新增、重命名、修改描述、删除
- 删除标签会重置相关交易并清理对应实例
- 删除标签后会追加记忆失效提示
- 新增 / 删除标签会导致分类预算整体失效
- 重命名标签会迁移分类预算对应键
- 旧参考设置页已有标签管理面板，但不能视为正式设置页已实现

原型设计要求：

- 标签管理必须明确区分“改名”“改说明”“删除”
- 删除和改说明都需要与重分类动作关系明确
- “其他”是系统兜底项，不应按普通标签允许删除
- 标签重命名必须作为独立能力直接暴露，不能只留“编辑说明”
- 当用户已删除全部自定义标签，或者账本初始就没有设置任何自定义标签时，标签列表中不应额外补一个“其他”条目凑数
- 上述空态下，列表区应改为明确空态提示，鼓励用户创建第一个标签

### 4.2.2 AI 记忆

AI 记忆是按账本维护的偏好规则快照，不是全局配置。

当前主仓库逻辑状态：

- `MemoryManager`、`SnapshotManager`、`LearningSession` 已形成可用链路

旧参考设置页已有能力：

- 查看当前记忆
- 逐条编辑
- 新增删除记忆条目
- 手动保存
- 查看快照历史
- 激活历史快照
- 删除非当前快照
- 手动触发立即学习

需要注意：

- 以上仅说明旧参考设置页里存在面板，不代表主仓库新的正式设置页表现层已经实现
- 原型阶段需要决定“记忆内容编辑”和“学习行为设置”是否分屏

本轮新增冻结口径：

- 记忆列表默认不支持在浏览态直接点条目进入编辑
- 需通过右下角“编辑记忆”按钮切换到编辑态
- 用户在编辑态按下回车时，表现层应立即在下方新增一个空白条目，并把焦点移动到新条目
- 新增条目后，列表序号必须即时重排
- 编辑期间不做自动保存；只有用户点击“完成”后才一次性提交当前草稿
- `AI 记忆概况` 中原先语义不清的“累计修正”卡，收敛为“实例库入口型指标”
- 该指标展示的核心含义固定为：**实例库距离上次学习的增量 / 实例库总量**
- 用户点击该指标后，应进入实例库相关入口或明细视图；不能再只是静态数字卡
- 点击“立即学习”成功后，必须给出明确成功反馈，并刷新 AI 记忆页与学习设置页的相关读模型

### 4.2.3 学习设置

学习设置与 AI 记忆相关，但不等于 AI 记忆本身。

当前应该纳入该分区的行为参数：

- 学习阈值 `learning.threshold`
- 自动学习 `learning.autoLearn`
- 收编阈值 `compression.threshold`
- 收编压缩比例 `compression.ratio`

当前实现结论：

- 主仓库逻辑层四项都已经通过 `ledger_prefs/{ledger}.json` 实现
- 旧设置页只接了“学习阈值”滑杆
- 自动学习开关、收编阈值、收编压缩比例仍无 UI
- 主仓库新的正式设置页表现层仍未实现

因此，交互原型不能再把“学习设置”简化成单个阈值滑杆，至少要把四项全部纳入结构设计。

本轮新增冻结口径：

- 学习阈值的最小可设值调整为 `1`
- AI 记忆页中的实例库增量指标与学习设置页中的学习阈值，需要基于同一账本上下文同步刷新

### 4.2.4 预算设置

预算设置是账本级配置，当前规格已冻结为：

- 月度总预算
- 分类预算

关键边界：

- 首页只消费预算读模型，不负责预算编辑
- 预算周期固定是当前自然月
- 预算卡不跟随首页 `data range`
- 分类预算与标签结构强联动

当前实现结论：

- 主仓库逻辑层已支持月度总预算与分类预算落盘
- 旧参考设置页已有预算面板
- 标签结构变更后分类预算会失效，并出现重新配置提示
- 主仓库新的正式设置页表现层仍未实现

### 4.2.5 重新分类

重新分类是设置页账本区的动作型能力。

它包含两类：

1. 标签操作后的渐进式重分类确认
2. 独立的“真全量重分类”入口
3. 按 7.1、7.2 页面树口径，S32 也参与全量重新分类链路，不应在正文其他地方被错误收窄为“只属于标签管理页”

当前实现结论：

- 标签新增 / 描述修改 / 删除后，主仓库已有渐进式确认逻辑与旧参考对话框
- 设置页账本区独立的“全量重分类”入口，在 v7 规格中已明确，但当前主仓库逻辑接口与新设置页表现层都未形成完整闭环

因此，设置页原型必须补出一个独立、清晰的“重新分类”区块，而不是把所有重分类行为都藏在标签管理内部。

本轮新增冻结口径：

- 全量重分类页底部风险提示统一改为：
  - `全量重分类，会清理未锁定交易的实例库记录。`
  - `并根据当前启用的记忆重新分类，请确认当前 AI 记忆情况，谨慎操作。`
- 该提示属于高风险动作说明，不得再沿用旧文案中语义模糊或仅强调耗时的版本

## 7. 设置页面树与流转

本章定义设置页原型的信息架构树。

约定：

- 顶层节点只允许两个区域节点：`全局区`、`账本区`
- 区域节点下是入口节点
- 叶子节点是具体设置功能 `Sxx`
- 动作型节点也必须在树中显式出现，不能只埋在弹窗里

### 7.1 设置页面树

```text
设置页 Root
├── 全局区
│   ├── AI 配置
│   │   ├── S1 AI 提供方选择
│   │   ├── S2 API Key
│   │   ├── S3 Base URL
│   │   ├── S4 模型选择
│   │   └── S5 AI 全局参数
│   ├── 自述
│   │   └── S8 自述
│   ├── 账本管理
│   │   ├── S9 账本切换
│   │   ├── S10 账本创建
│   │   ├── S11 账本重命名
│   │   └── S12 账本删除
│   ├── 数据导出
│   │   └── S14 数据导出
│   └── 关于
│       └── S13 关于
└── 账本区
    ├── 标签管理
    │   ├── S15 标签新增
    │   ├── S16 标签重命名
    │   ├── S17 标签描述修改
    │   ├── S18 标签删除
    │   └──── 
    |       └──S32 标签变更后的渐进式重分类
    ├── AI 记忆
    │   ├── S19 AI 记忆查看
    │   ├── S20 AI 记忆编辑
    │   ├── S21 立即学习
    │   ├── S22 记忆快照历史
    │   ├── S23 快照回退
    │   └── S24 快照删除
    ├── 预算设置
    │   ├── S29 月度总预算
    │   └── S30 分类预算
    ├── 学习设置
    │   ├── S25 学习阈值
    │   ├── S26 自动学习开关
    │   └── S27 收编阈值
    └── 全量重新分类
        └── S31 全量重分类入口
            └──S32 标签变更后的渐进式重分类
```

### 7.2 页面流转定义

推荐采用“一级总览 + 二级详情页”的稳定流转：

1. 首页或底部导航进入 `设置页`
2. `设置页`分上下全局/账本区域，两个区域显示一级入口节点列表
3. 点击区域内某个入口节点，进入对应二级页面
4. 二级页面内部再承载对应 `Sxx` 叶子节点
5. 叶子节点若需要危险确认、测试连接、批量执行，使用对话框或底部抽屉，不再继续下钻三级页面

### 7.3 根页面职责

`设置页 Root` 只负责：

- 展示当前账本上下文
- 展示全局区与账本区两个区域
- 展示每个入口节点的摘要信息
- 承担区域内导航，不直接编辑复杂表单
- 底部三栏导航中的“设置”必须处于清晰激活态，反馈强度与记账页激活态一致

`设置页 Root` 不负责：

- 直接承载大段表单
- 在同一页面混放所有设置字段
- 把危险动作和常规表单塞进同一滚动画布而没有层级

### 7.4 全局区流转

`设置页 Root`
→ `全局区`
→ 入口节点之一：

- `AI 配置页`
- `自述页`
- `账本管理页`
- `数据导出页`
- `关于页`

叶子节点归属：

- `AI 配置页`：S1-S5
- `自述页`：S8
- `账本管理页`：S9-S12
- `数据导出页`：S14
- `关于页`：S13

### 7.5 账本区流转

`设置页 Root`
→ `账本区`
→ 入口节点之一：

- `标签管理页`
- `AI 记忆页`
- `预算设置页`
- `学习设置页`
- `全量重新分类页`

叶子节点归属：

- `标签管理页`：S15-S18，以及通往 S32 的标签变更触发链路
- `AI 记忆页`：S19-S24
- `预算设置页`：S29-S30
- `学习设置页`：S25-S28
- `全量重新分类页`：S31，并按 7.1 页面树承接 S32 所在的重分类链路

### 7.6 跨页面联动规则

以下联动必须在页面树上被显式考虑：

1. `标签管理页` 修改标签后，可能影响 `预算设置页`
2. `标签管理页` 与 `全量重新分类页` 都会触发重分类相关动作；S32 的链路归属以 7.1、7.2 为准，正文其他章节不得擅自改单归属
3. `AI 记忆页` 与 `学习设置页` 强相关，但必须拆页，避免内容编辑和行为参数混在一起
4. `账本管理页` 切换账本后，账本区所有入口页面都要切换上下文

## 8. 参考子仓库对设置原型的价值与边界

参考子仓库 `Moni-UI-Prototype/` 在设置相关信息上提供了两类价值：

### 8.1 可继承的稳定口径

- 设置页分为全局区与账本区
- 全局区：AI 配置 / 自述 / 账本管理 / 数据导出 / 关于
- 账本区：标签管理 / AI 记忆 / 预算设置 / 学习设置 / 全量重新分类
- 首页不承担复杂配置编辑

### 8.2 当前不能直接照搬的部分

- 原型仓库当前主要完成的是首页与记账页，不是新的设置页原型
- 其设置页相关内容多数仍停留在文字层级，没有形成新的交互原型
- 主仓库旧 `SettingsPage.tsx` 虽有实现，但其 IA 仍带有旧 PixelBill 风格，不应直接等同于 Moni 当前最终设置页信息架构

结论：

- 设置页原型需要以本文件为统一总表重新设计
- 不应直接照抄旧设置页实现
- 也不应只依赖 `DESIGN.md` 的两行栏目描述

## 9. 可交互原型设计的直接指导

为避免后续原型设计再次失焦，本轮设置页原型应至少覆盖以下区块：

### 9.1 必做区块

1. 全局区
2. 账本区
3. 每个区块内的项级说明
4. 二级页面具体视觉和交互设计
5. S32 所在重分类链路与入口关系，渐进式UI链路完备设计

### 9.2 必做设置项

1. AI 配置
2. 自述
3. 账本管理完整 CRUD
4. 标签管理
5. AI 记忆
6. 学习设置四项
7. 预算设置
8. 全量重新分类
9. 关于（入口节点）

### 9.3 可后置项

1. 数据导出页的具体展开内容
2. 语言
3. 主题
4. 关于（二级页面内容）

### 9.4 设计时必须避免的误判

1. 不要把“账本切换”误当成“账本管理”全部能力
2. 不要把“学习阈值滑杆”误当成完整“学习设置”
3. 不要把“标签管理中的重分类弹窗”误当成完整“全量重新分类系统”，也不要忽略 7.1、7.2 中 S32 的链路位置
4. 不要把“旧设置页已有组件”误当成当前信息架构已经定稿

## 10. 本文结论

Moni 当前的设置系统，已经在底层实现了大量真实能力，但规格与入口仍然分散。

本轮统一梳理后，可以得到三个确定结论：

1. 设置系统的正式骨架应为“全局区 + 账本区”
2. 当前最大缺口不是底层能力，而是新的正式设置页表现层与信息架构尚未实现
3. 下一步设置页可交互原型，应以本文件为总表，以预算规格和 v7 记忆规格作为专项补充，而不是直接照搬旧 `SettingsPage.tsx`
