# Moni UI / UX DESIGN.md

## 0. 文档定位

本文件是 Moni 当前阶段唯一有效的 UI / UX 设计标准。

执行规则如下：

1. 后续所有 UI 原型设计、前端实现、交互调整、视觉修订，均以本文件为唯一准绳。
2. 若本文件与其他文档、历史原型、聊天记录、旧设计稿、过往第三方设计工具产物发生冲突，以本文件为准。
3. 若后续需要修改已定稿的交互或视觉规则，必须先更新本文件，再进行代码实现。
4. 其他文档的职责如下：
   - `Moni_Requirements_v2.md`：功能需求参考
   - `Moni_Brand_Design_Spec.md`：品牌视觉与 SVG 资产参考
   - `Moni_Stitch_Guide_and_Prompts.md`：历史设计探索记录，当前阶段不作为执行依据
5. 本文件只收录已经明确确认、可直接执行的 UI / UX 结论；未确认事项不作为实现依据。
6. 当前阶段的原型路线为：直接产出可交互原型并在本仓库中迭代，不再依赖 Stitch 参与当前 UI / UX 方案生成。
7. 当前产品运行环境以移动端为准：实际产品将通过 Capacitor 运行在 Android 上，原型与交互实现必须优先服务移动端触控模式。
8. 桌面浏览器中的 F12 移动端模拟模式，属于当前原型开发阶段的有效测试环境；在该模式下，滑动、长按、拖拽等核心手势也必须可正常验证。

## 1. 产品定位

Moni 是一个越用越聪明的 AI 记账助手。

它的核心不是“让用户认真维护账本”，而是：

- 导入账单
- 浏览流水
- 发现分错时改一下
- AI 从这次修正中学会

三条顶层哲学：

1. AI 是学生，用户是老师。每一次分类修正都是一堂课。
2. 记账是结果，不是操作。用户不应该被迫多做无意义的维护动作。
3. 财务是私密且个性化的。AI 理解的是“这个人”，不是抽象类别。

首页不是仪表盘页，也不是单纯的流水页。首页是主舞台：顶部看板先传达整体感受，下方流水承接最重要的用户行为——看钱花到哪里、顺手纠错、教会 AI。

## 3. 视觉基调

### 3.1 风格方向

- 风格：轻量 Memphis
- 气质：温暖、活泼、轻松，但不幼稚
- 原则：有情绪、有个性，但金融信息必须清晰可读

### 3.2 基础视觉规则

- 页面背景：暖米白
- 首页手机原型幕布尺寸标准为 390 × 860
- 首页原型默认按移动端单手触控阅读与操作方式组织，不以桌面 hover 交互为前提
- 主卡片：白底、2px 深色描边、14-16px 圆角
- 次级卡片：白底、1.5px 浅灰描边、10-12px 圆角
- 金额和关键数字：等宽数字风格
- 页面上允许少量 Memphis 几何印花，但不能干扰信息阅读

### 3.3 Memphis 印花规则

- 全局装饰采用程序生成，而不是手动硬编码摆点
- 由一个可复用的随机渲染函数统一控制
- 随机不等于混乱，要求“随机生成、结果稳定”
- 单屏密度低，优先出现在边角、卡片缝隙、留白区域
- 不允许压住文字、金额、按钮或图表

## 4. 品牌元素分工

### 4.1 Header 标题

- Header 使用完整的 `Moni` 字标，不是单独的 M 图标
- Header 中的 M 不做猫耳图标化处理
- 字标右上区域固定保留品牌三色装饰簇
- 三色装饰必须聚在 M 与 oni 的交界上方，不能偏到最右侧
- Header 始终粘性固定在顶部，不随无限滚动消失

### 4.2 底部中央按钮

- 底部中央按钮使用独立的品牌 icon 语汇
- 它承担品牌、首页入口、AI 状态、AI 控制四种职责
- 中央按钮里的 M 可以是猫耳感更强的版本
- M 与三色装饰应整体比当前版本更大、更有存在感

### 4.3 App Icon

- App Icon 与底部中央按钮整体上可复用同一套核心设计
- 初版允许在中央按钮 SVG 基础上做少量适配后生成 PNG

### 4.4 三色系统

品牌三色固定为：

- 珊瑚圆
- 天蓝圆
- 向日黄菱形

这是所有品牌触点与 AI 工作态特效的统一视觉签名。

## 5. 全局导航结构

底部固定三栏导航：

- 左：设置
- 中：首页主入口 / 品牌按钮 / AI 状态 / AI 控制
- 右：记账

导航原则：

- 中间按钮是绝对主入口，视觉显著高于两侧
- 两侧图标必须使用同一套描边语言，不抢中间按钮视觉权重
- 设置与记账是工具入口，不是主舞台

## 6. 首页总体结构

首页是一个单一垂直滚动画布，从上到下固定为：

1. Header
2. 看板显示卡
3. 情景提示卡
4. 统计摘要栏
5. 分类概览卡片
6. 分类标签轨道
7. 按天分组的流水卡片列表
8. 底部三栏导航

Header 始终固定在顶部。

当用户向上滚动时：

- Header 保持原位
- 分类标签轨道会吸附到 Header 下方
- 标签轨道以上区域被推出屏幕
- 流水区域成为主阅读区并持续向下滚动

---

## 7. Header

Header 左侧是 Moni 字标，右侧是账本选择器。

要求如下：

- 默认账本名为“日常开销”
- 账本选择器在首页只负责切换，不负责复杂管理
- 账本管理入口归属设置页全局区
- Header 是全局粘性元素，首页滚动、流水无限滚动时都保留在顶部

## 8. 顶部看板显示卡

### 8.1 角色定义

顶部看板不是单一预算卡，而是一个“显示卡容器”。

它的任务不是做复杂数据分析，而是让用户在第一眼获得消费节奏感。

### 8.2 卡片组成

有预算时：

- 卡片 A：预算卡
- 卡片 B：近 7 天支出折线图卡

无预算时：

- 只显示折线图卡
- 不显示轮播圆点
- 不显示预算卡

### 8.3 轮播规则

- 看板卡片切换方向是上下切换，不是左右切换
- 自动轮播周期：30 秒
- 用户手动切换看板后，自动轮播暂停 5 分钟
- 在移动端触控模式下，上下切卡与页面纵向滚动需要做明确手势区分，不能互相误触

### 8.4 预算卡规则

- 展示当月预算、已用金额、剩余金额、剩余天数、日均可用金额
- 顶部进度条贯穿整卡
- 预算状态直接影响整卡气质
  - 健康：薄荷绿
  - 警戒：琥珀橙
  - 超支：珊瑚红

### 8.5 折线图卡规则

- 默认显示最近 7 天
- 视觉必须轻盈，不做厚重图表
- 不显示 Y 轴
- 底部显示日期标签
- 折线图卡内部支持左右滑动切换历史时间窗口
- 看板本身负责上下切换卡片；时间窗口切换只发生在折线图卡内部
- 在移动端与 F12 移动模拟模式下，折线图卡的横滑翻窗必须可稳定触发，不能被外层看板切卡或页面滚动抢走

## 9. 情景提示卡

情景提示卡位于看板与统计摘要之间。

规则如下：

- 有内容时显示，无内容时整块区域消失
- 它是自然插入的临时卡片，不是弹窗，不打断当前阅读
- 可承载的典型内容：
  - 距上次导入已 N 天
  - 本周消费偏高
  - AI 学到新的偏好
  - 首次导入后对 AI 控制入口的提示

---

## 10. 统计摘要栏

统计摘要栏固定为三张小卡：

- 支出
- 收入
- 笔数

规则如下：

- 标题前缀必须与 Data Range 联动
- 例如：本月支出 / 本周支出 / 今日支出 / 区间支出
- 该区域只提供快速理解，不承担深度分析任务

## 11. 分类概览卡片

### 11.1 表现方式

- 使用横向堆叠条
- 不使用饼图
- 图例与横条信息一一对应
- 分类概览中应保留所有已出现的分类
- 配色应保持轻盈、明快、低负担，可以使用柔和彩色，但不能因为克制而牺牲分类可区分性

### 11.2 未分类规则

- 横条末尾必须保留未分类占位段
- 未分类段必须使用半透明或斜杠阴影表达
- 不允许把未分类画成普通实心色块

### 11.3 Data Range Picker 入口

- 右上角是 Data Range Picker 的唯一入口
- 点击后打开居中的二级面板
- 提供快捷项：
  - 今天
  - 本周
  - 本月
  - 近三月
- 同时支持自定义范围
- 自定义范围保留现有快捷项，不允许因为增加滑条而移除快捷选择

### 11.4 联动规则

Data Range Picker 改变后，必须同时联动：

- 统计摘要栏标题
- 分类概览卡片右上角文本
- 分类概览横条与图例
- 流水列表中的数据范围

如果使用快捷范围：

- 右上角显示快捷项名称

如果使用自定义范围：

- 右上角显示完整区间
- 统计摘要使用“区间支出 / 区间收入 / 区间笔数”命名

### 11.5 自定义范围交互

- 自定义范围使用双档位 Range Slider
- 左档位控制起始日期，右档位控制结束日期
- 两个档位只允许水平移动
- 起始档位不得超过结束档位，结束档位不得早于起始档位
- 拖动过程中，起止日期文字必须实时更新
- 自定义范围同时支持鼠标拖拽与触摸滑动
- 面板中保留可直接编辑的起始 / 结束日期输入
- 起始日期不得早于全局最小日期
- 结束日期不得晚于全局最大日期
- 结束日期按完整一天处理，确保当天数据全部计入
- 面板底部显示 MIN / MAX 边界提示
- 点击背景遮罩关闭面板

## 12. 分类标签轨道

### 12.1 定位

分类标签区域不是简陋的分页 pills，也不是纯粹的左右切页器。

它的目标交互语义是：

- 一个可横向浏览的标签轨道
- 最终理想形态是无限轮盘

### 12.2 当前执行标准

- 正式目标：无限轮盘
- 原型阶段允许先实现有限轮盘
- 即便是有限轮盘，也必须表现为“横向轨道浏览”
- 不允许退化成静态分页窗口

### 12.3 视觉规则

- 标签维持胶囊形态
- 显示文案继续使用中文标签
- 激活态：深色填充
- 未激活态：白底描边
- 未分类：警示态，视觉上更醒目
- 吸附后必须稳定固定在 Header 下方
- 向下滚动时标签轨道不能被继续推出屏幕或消失

### 12.4 文案与数据分离规则

- 分类数据结构中的键，当前阶段直接使用中文
- UI 标签直接显示这个中文键
- 不再额外维护一套英文 key 到中文显示文案的映射关系
- 若存在更长的分类描述文本，它只作为说明信息存在，不进入标签轨道主显示
- 胶囊标签的视觉表达可以保留，但不能借此退回分页式实现

## 13. 默认分类体系与图标系统

### 13.1 默认分类

默认分类固定为以下 11 类：

- 正餐：日常正餐支出（早午晚），如快餐、正餐、工作餐
- 零食：零食、饮品、小吃等非正餐食品
- 交通：公共交通、打车、加油、停车等出行费用
- 娱乐：电影、游戏、演出、会员订阅等娱乐消费
- 大餐：聚餐、大餐、宴请、高档餐厅等特殊餐饮
- 健康：医疗、药品、保健品、健身器材等健康支出
- 购物：日用品、服装、电子产品、网购等购物消费
- 教育：书籍、课程、培训、考试等教育支出
- 居住：房租、水电煤、物业、维修等居住费用
- 旅行：旅游、酒店、机票、景点门票等旅行支出
- 其他：其他未分类支出

### 13.2 图标系统

- 每个默认分类都属于一个轻量级图标组
- 一个分类允许对应多个图标变体
- 例如：
  - 正餐 可区分堂食、外卖、便当
  - 交通 可区分地铁、打车、加油
  - 旅行 可区分机票、酒店、门票
- 原型中每个类别至少应体现出一个图标变体
- 未分类始终使用问号图标

## 14. 流水列表

### 14.1 基础结构

- 按天分组，一天一张卡
- 按时间倒序
- 今天、昨天使用语义标签
- 更早日期显示为 M月D日，不带年份

### 14.2 初始折叠规则

页面停留在初始原位时：

- 只有最近一天展开
- 其他日期卡片默认收起
- 若某一天全部已分类，则收起态表现为单行摘要
- 收起态应更轻：白底、浅描边、轻透明感，不应与展开态卡片拥有相同重量

### 14.3 滚动阶段规则

首页滚动存在三个阶段：

#### 阶段 A：页面原位

- 仅最近一天展开
- 其他日期收起

#### 阶段 B：开始上滑，但看板区域尚未完全退出

- 在这一小段滚动区间内
- 屏幕上当前可见的、原本收起的日期卡片，需要自动展开
- 展开必须带动画
- 这是从“首页浏览态”过渡到“流水主场态”的关键体验

#### 阶段 C：标签轨道已吸附到顶部后继续向下

- 所有日期默认展开
- 流水进入持续向下浏览状态

### 14.4 条目内容显示规则

在“全部”视图下：

- 若存在 user category：
  - 显示分类标签
  - 不显示 AI 字段
- 若不存在 user category：
  - 显示 AI 给出的分类标签
  - 同时显示 AI reasoning
  - 用户应能直观看出“这是 AI 的暂时理解，尚未被确认”

在具体分类视图下：

- 可隐藏 category tag，避免冗余
- 可保留 AI reasoning，帮助用户理解 AI 判断依据

### 14.5 视觉提示规则

- 已分类条目：使用分类对应浅底图标块
- 未分类条目：使用问号图标 + 警示态标签 + 更强的“未完成感”
- AI 推理是解释，不是主标签
- user category 与 AI reasoning 必须在语义上被严格区分

## 15. 条目交互

### 15.1 单击

- 单击条目进入交易详情页
- 详情页从右侧滑入

### 15.2 长按拖拽分类

- 长按条目后，条目被“拎起”
- 整屏出现半透明蒙版
- 交易列表以上区域出现两列分类投放格
- 手指悬浮到分类格时，对应格子高亮或放大
- 松手投放后进入“可选描述输入”
- 完成输入或跳过后，蒙版消失，条目完成更新
- 该交互以移动端单指触控为准，不依赖桌面 hover 才能完成
- 在桌面浏览器 F12 的移动模拟模式下，也必须能够完成“长按—拖动—悬停高亮—松手投放”的完整链路

拖拽交互的目标气质是：

- 有点游戏感
- 有触感
- 有收尾反馈
- 但不能花哨得像小游戏

## 17. AI 工作态与控制

### 17.1 工作态联动

当 AI 正在处理某一天的数据时：

- 该日卡片黑色边框切换为品牌三色流光带
- 卡片内未完成条目显示骨架屏
- 底部中央按钮同步出现同款流光 / 发光
- 中央按钮的流光节奏必须与被处理日卡片保持同步
- 不允许只用单一蓝色常亮代替

### 17.2 中央按钮控制交互

当前定稿交互不是两个独立按钮，而是一条向上弹出的开关条。

规则如下：

- 长按中央按钮后，向上弹出纵向开关条
- 手指保持按住并向上滑动时，进入开启态反馈
- 手指滑到下半区域并松手时，进入关闭态反馈
- 交互应当像一个连续的滑控动作，而不是两个傻按钮供用户点选

### 17.3 软停止机制

关闭 AI 时采用软停止，而不是硬中断。

规则如下：

- 已经发出的请求不应被立刻打断
- 当前处理批次应先完成写入

1. 用户打开后，是否能立刻看懂当前消费状态？

## 23. 手势交互实现规范（Capacitor Android WebView 适配）

### 23.0 背景与目标

Moni 运行在 Capacitor Android 端，底层是 Android WebView（Chromium 内核）。开发测试在浏览器 F12 移动设备模式下进行。两个环境在手势处理上存在系统性差异，特别是指针捕获、嵌套滚动、长按行为、拖拽事件四个方面。

本节的目标是：通过统一的 CSS 防御层和手势实现规范，使交互行为在 F12 模拟和真机 WebView 之间保持一致，消除"F12 里正常、真机上崩坏"以及"F12 里就坏了"两类风险。

---

### 23.1 全局 CSS 防御层

以下 CSS 规则必须在全局样式中设置，不得遗漏：

```css
/* 1. 禁用双击缩放 — 消除 300ms tap delay */
html {
  touch-action: manipulation;
}

/* 2. 禁用长按系统菜单 — 防止长按时弹出文本选择/链接预览 */
* {
  -webkit-touch-callout: none;
  -webkit-user-select: none;
  user-select: none;
}

/* 3. 对需要文本选择的区域单独恢复 */
input, textarea, [data-selectable] {
  -webkit-user-select: text;
  user-select: text;
}

/* 4. 防止嵌套滚动穿透 */
[data-scroll-container] {
  overscroll-behavior: contain;
}

/* 5. 禁用 Android WebView 的 overscroll 发光效果 */
body {
  overscroll-behavior: none;
}

/* 6. 禁用默认拖拽行为 */
img, a, svg {
  -webkit-user-drag: none;
}
```

**强制要求**：以上 6 条规则写入全局 CSS，不允许在组件层面逐一添加。

---

### 23.2 核心原则：指针捕获（Pointer Capture）

**这是当前代码中多个 bug 的共同根因，必须理解后再做任何手势修改。**

在 Pointer Events 模型中，当 `pointerdown` 发生在元素 A 上时，浏览器会对元素 A 进行隐式指针捕获（implicit pointer capture）。这意味着后续所有 `pointermove` 和 `pointerup` 事件都会被送到元素 A，而不是指针当前悬停的元素 B。

**推论**：

1. 如果你在 `pointerdown` 之后动态渲染了新元素（如弹出控制条），新元素的 `onPointerMove` 不会被触发，因为指针被原元素捕获了
2. 需要跨元素追踪指针位置时，必须使用以下方案之一：
   - 在已捕获指针的元素上监听 `pointermove`，用 `clientX/clientY` + `document.elementFromPoint()` 判定命中
   - 使用 `window.addEventListener("pointermove", ...)` 全局监听
   - 使用 `element.releasePointerCapture(pointerId)` 主动释放捕获

---

### 23.3 各交互点的具体实现约束

#### 23.3.1 看板卡片上下轮播（第 8 节）

**当前实现方式**：手写 Pointer Events + ref 记录起止坐标 + 手动方向锁
**评估**：基本可用，但方向锁定逻辑脆弱。未来建议迁移到 Framer Motion `drag="y"`。
**当前阶段要求**：保持现有实现，修复以下问题即可：

- 在看板容器上添加 `touch-action: pan-x`（允许水平穿透，拦截垂直，防止与页面滚动冲突）

#### 23.3.2 折线图卡内左右滑动（第 8.5 节）

**当前实现方式**：手写 Pointer Events + `stopPropagation` 防止冒泡到看板层
**评估**：与看板的嵌套关系通过 `stopPropagation` 处理，基本可用。
**当前阶段要求**：保持现有实现，添加以下防御：

- 折线图容器添加 `touch-action: pan-y`（允许垂直穿透，拦截水平）

#### 23.3.3 分类标签轨道横向滚动（第 12 节）

**当前实现方式**：`overflowX: "auto"` 原生滚动
**评估**：正确，无需使用 drag 手势。
**当前阶段要求**：

- 容器添加 `overscroll-behavior-x: contain`（防止滚到头时触发页面滚动）
- 确认 sticky 吸附后横向滚动仍正常（不被 sticky 容器截断）

#### 23.3.4 长按条目 → 拖拽分类（第 15.2 节）

**当前实现方式**：

- 长按判定：`onPointerDown` + `setTimeout(420ms)` + 移动超 8px 取消
- 拖拽追踪：`window.addEventListener("pointermove/pointerup", ...)`（全局监听，正确！）
- 命中判定：`document.elementFromPoint()` + `data-drop-category` 属性

**评估**：拖拽阶段的全局监听是正确的。但存在一个问题：

**BUG — `onPointerLeave` 提前取消长按**：
条目元素上同时绑定了 `onPointerLeave={onItemPointerUp}`。在 F12 移动模式下，手指微小抖动可能导致 pointer 暂时离开元素边界，触发 `pointerleave`，进而取消长按定时器。

**修复方案**：

- 移除条目的 `onPointerLeave` 处理
- 长按取消只依赖两个条件：移动距离 > 8px 或 `pointerup` 提前触发
- 拖拽激活后锁定 body 滚动：`document.body.style.overflow = "hidden"`，拖拽结束后恢复

#### 23.3.5 长按中央按钮 → AI 控制滑条（第 17.2 节）

**当前实现方式**：

- 长按判定：`onPointerDown` → 420ms 定时器 → `setControlOpen(true)`
- 命中追踪：控制条自身的 `onPointerMove` → `updateControlHit(clientY)`
- 执行：`onPointerUp` → 检查 `controlHit` → 开启/关闭

**BUG — 控制条的 `onPointerMove` 从不触发**：

根因就是 23.2 节说的指针捕获问题。`pointerdown` 发生在中央按钮的父容器 div 上，该 div 隐式捕获了指针。之后动态渲染的控制条子元素的 `onPointerMove` 永远不会收到事件。`controlHit` 始终为 null，松手时什么都不执行。用户被迫再单击一次才能触发——但这并不是设计意图。

**修复方案**（选其一）：

**方案 A（推荐，最小改动）**：在父容器的 `onPointerMove` 中统一处理

```jsx
// 父容器 div 增加 onPointerMove
<div
  style={{ touchAction: "none" }}
  onPointerDown={onStartControl}
  onPointerMove={(e) => {
    // 只在控制条打开时追踪
    if (controlOpen) {
      onUpdateControlHit.move(e.clientY);
    }
  }}
  onPointerUp={onEndControl}
  onPointerCancel={onCancelControl}
>
```

控制条自身不再需要 `onPointerMove`。指针捕获在父容器上，父容器的 `onPointerMove` 能正常收到事件。

**方案 B（更彻底）**：使用全局监听，与拖拽分类同模式

```javascript
useEffect(() => {
  if (!controlOpen) return;
  const handleMove = (e) => updateControlHit(e.clientY);
  const handleUp = () => { handleEndControl(); };
  window.addEventListener("pointermove", handleMove);
  window.addEventListener("pointerup", handleUp);
  return () => {
    window.removeEventListener("pointermove", handleMove);
    window.removeEventListener("pointerup", handleUp);
  };
}, [controlOpen]);
```

**额外 BUG — `onPointerLeave` 取消控制**：
父容器还绑定了 `onPointerLeave={onCancelControl}`。当手指从按钮滑到控制条时，如果控制条的绝对定位区域超出了父容器的边界，`pointerleave` 会触发，直接关闭控制条。

**修复**：移除 `onPointerLeave={onCancelControl}`。控制条关闭只通过 `pointerup`（有效操作）或点击外部区域（取消操作）来触发。

#### 23.3.6 主页面纵向滚动与嵌套手势的协调

总原则：

1. 主页面使用原生 `overflow-y: auto` 滚动，不使用 drag 手势
2. 看板区域的垂直切换和折线图的水平滑动，由当前的手写 pointer 事件管理，通过 `touch-action` 属性分工
3. 长按拖拽激活时锁 body scroll，结束后恢复
4. 标签轨道横向滚动是原生 CSS 滚动，`overscroll-behavior-x: contain` 防穿透
5. 未来条件允许时，看板和折线图的手势迁移到 Framer Motion `drag`

### 23.6 禁止事项

1. **禁止**在动态弹出的子元素上依赖 `onPointerMove` 追踪指针（指针已被父元素捕获）
2. **禁止**使用 `onPointerLeave` 作为关闭/取消手势的手段（指针抖动和隐式捕获会导致误触发）

## 24. 设置页 UI/UX 设计

本章定义 Moni 设置页的完整视觉与交互标准。详细数值规格见 `Moni_Settings_UIUX_Spec_v1.md`，可交互原型参考见 `MoniSettingsPrototype.jsx`。

### 24.1 总体结构

设置页采用 **"一级总览 + 二级详情页"** 的稳定流转结构：

1. 设置页 Root 展示全局区与账本区两个 SectionCard
2. 点击 SectionCard 内的行项，进入对应二级页面
3. 二级页面内部使用 Dialog（居中弹窗）或 BottomSheet（底部抽屉）承载叶子级操作
4. 不存在三级页面

设置页手机原型与首页/记账页共用同一手机框架尺寸（390 × 860），共用同一套设计 token、品牌 Logo、底部导航栏。

### 24.2 Root 页面

Root 页面不承载任何表单编辑，只负责：

- 展示品牌 Logo + "设置" 页标题
- 展示全局区 SectionCard（AI 配置 / 自述 / 账本管理 / 数据导出 / 关于）
- 展示账本区 SectionCard（标签管理 / AI 记忆 / 预算设置 / 学习设置 / 全量重新分类）
- 每个行项展示 emoji 图标、标签名、摘要描述、当前值、右箭头

SectionCard 视觉规格：

- 白底、2px 深色描边、16px 圆角（主卡片级别）
- 内部行项通过 SettingRow 组件渲染，底部 0.5px 浅灰分隔线

全局区标题格式：`🌐 全局配置`
账本区标题格式：`📒 账本：{当前账本名}`，右侧副标题"当前账本配置"

### 24.3 二级页面通用规范

#### 页头 SubPageHeader

- 左侧返回箭头 + 页标题
- 粘性固定在顶部
- 返回箭头点击回到 Root
- 二级页中的保存 / 编辑 / 新建 / 新增 / 历史版本 / 立即学习等动作，不放在 Header 右上角，统一放在对应内容块底部或内容区内部

#### 保存反馈

- 所有需要保存的页面（自述、预算、学习设置）使用内容块底部动作按钮
- 保存成功弹出 Toast 提示，2 秒后自动消失

#### 危险操作确认

- 删除账本、删除标签、全量重分类均需要 Dialog 确认
- 确认弹窗使用 danger 变体按钮（pinkBg + coral 文字）
- 弹窗文案必须明确说明影响范围和不可恢复性

### 24.4 弹窗与抽屉

设置页中使用两种覆盖层组件：

**Dialog（居中弹窗）**：用于确认、输入、选择

- 宽度 85%，最大高度 75%
- 圆角 18px，描边 2px solid C.dark
- 点击遮罩关闭

**BottomSheet（底部抽屉）**：用于列表浏览（快照历史、锁定交易列表）

- 从底部滑出，顶部圆角 20px
- 带拖拽指示条（36×4 圆角条）
- 最大高度 70%，内容区可滚动

### 24.5 标签管理与渐进式重分类（S32）

这是设置页中交互复杂度最高的区域。标签的三种变更操作（新增、修改描述、删除）各自触发不同的渐进式重分类 UI 链路：

标签列表层的动作收敛如下：

- 列表行右侧保留统一的“编辑”入口与“删除”入口
- “编辑”进入统一 Dialog，但必须在 Dialog 内显式拆分为“重命名键”和“编辑描述”两个互斥标签页
- 不允许退回到“只能编辑说明、无法直接改名”的旧形态
- 不允许把“改名 / 改说明 / 删除”三个动作直接平铺在列表行里，避免压缩标签文本可读性

#### 新增标签后

```
→ Dialog："标签「xxx」已新增，需要开始分类吗？"
    → [暂时跳过]     → 结束
    → [现在启动分类]  → 结束（仅通知消费端）
```

#### 删除标签后

```
→ Dialog："选择重分类范围"
    → [仅受影响的交易]          → 结束（对受影响日期入队）
    → [全账本所有未锁定交易]    → BottomSheet：锁定交易解锁确认
        → 勾选要解锁的条目 → [确认并开始重分类] → 结束
    → [稍后处理]               → 结束
```

#### 修改描述后

```
→ Dialog："标签定义已更改，需要重新分类吗？"
    → [暂时跳过]               → 结束
    → [该标签下仅未锁定交易]    → 结束（对该标签范围入队）
    → [该标签下所有交易]        → BottomSheet：锁定交易解锁确认
        → 勾选 → [确认并开始重分类] → 结束
```

#### 锁定交易列表 BottomSheet

- 标题："以下锁定交易将被包含"
- 说明文案："勾选后将临时解锁并参与本次重分类"
- 每行：checkbox + 交易摘要（日期 商户 金额） + "已锁定"橙色标签
- 底部：取消 / 确认并开始重分类

### 24.6 AI 记忆页

AI 记忆页包含四个功能层：

1. **状态指标**：实例库入口型指标（greenBg）+ 记忆条数（blueBg）
2. **操作入口**："立即学习"+ "历史版本"
3. **记忆列表**：浏览态与编辑态分离的有序列表编辑器
4. **快照历史**：BottomSheet 展示全部快照，支持回退和删除

实例库指标规则：

- 含义固定为“距离上次学习的增量 / 实例库总量”
- 该卡片本质是可点击入口，不再使用“累计修正 / 阈值”的旧语义
- 当前阶段只冻结入口语义，不在设置页文档中臆造更深的实例库字段结构

记忆列表规则：

- 浏览态下，点击条目本身不能直接进入编辑
- 通过内容块底部“编辑记忆”按钮进入编辑态
- 每条右侧出现 × 删除按钮（coral 色）
- 编辑态按回车会在下一行新增空白条目，并立即重排序号
- 编辑期间不自动保存；点击“完成”后才一次性提交当前草稿并退出编辑态

### 24.7 预算设置页

- 月度总预算使用大字号 input（fontSize 22），¥ 前缀
- 分类预算以标签列表形式展示，每行一个 input
- 合计值实时计算并显示
- 超出月度总预算时：合计文字变红、容器边框变 pinkBd

### 24.8 学习设置页

三项参数独立卡片：

- 自动学习：Toggle 开关（C.mint 激活色）
- 学习阈值：Range Slider（C.mint），当前值大字显示
- 收编阈值：Range Slider（C.amber），当前值大字显示

每张卡片包含标题、说明文字和操作控件。

补充冻结口径：

- 学习阈值最小值为 `1`
- “立即学习”成功后，学习设置页中的实例库增量显示需要与 AI 记忆页同步刷新

### 24.9 全量重分类页

- 说明卡（warmBg）解释重分类含义
- 账本概况卡：总交易数 / 未锁定数（amber）/ 已锁定数（mint）
- 红色警告条
- "开始全量重新分类" danger 按钮 → Dialog 确认
- 运行中按钮变 disabled + Toast 提示

红色警告条文案固定为：

- `全量重分类，会清理未锁定交易的实例库记录。`
- `并根据当前启用的记忆重新分类，请确认当前 AI 记忆情况，谨慎操作。`

### 24.10 设置页 Memphis 装饰

- Root 页面使用 seed=555 生成装饰
- 二级页面使用 seed=777，避免视觉雷同
- 装饰层 zIndex 0，内容层 zIndex 1
- 装饰密度与首页一致（每屏 6 个元素）

### 24.11 底部导航态

设置页处于激活态时：

- 左侧齿轮图标：描边加粗（1.8），深色，文字加粗
- 中央品牌按钮：保持常态
- 右侧记账图标：保持浅色非激活态

这与首页/记账页各自激活时的导航态处理方式一致。
